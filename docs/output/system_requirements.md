# システム要件定義書

## システム概要

**「迷わせないレコメンド」** は、自動車メーカー本部長向けショールームデモとして設計された、Knowledge Graph ベースの説明可能レコメンド体験である。SDV 時代に増大する機能・サービス・オプションの選択肢に対し、生活者の判断負荷（メンタルパフォーマンス）を軽減し、販売店が「なぜその提案か」を説明できる状態を作ることを目的とする。

**1. システムの目的**

- **解決する課題**: 生成 AI 単体では推薦理由がブラックボックス化しやすく、販売店の属人的スキルに依存した商談が続く。生活者・販売店双方が「何を選べばよいか」わからない選択過多の状態。
- **目標とする成果**: 7〜10 分のデモフローで、①選択負荷の課題認識 → ②ユーザー理解 → ③AI との距離感の選択 → ④KG による説明可能性の体験 → ⑤3 候補への絞り込み → ⑥販売店トーク資産化 → ⑦AXIS 連携の示唆、を一気通貫で見せる。
- **期待されるビジネスインパクト**: OEM 向けに「説明責任を満たす推薦インフラ」としての差別化を印象づけ、販売店改革（商談品質・時間短縮・後悔率低減）への投資判断材料を提供する。

**2. 主な機能**

- **コア機能**
  - 7 画面デモフロー（Opening → Quick Questions → AI Delegation → KG Visualization → Recommendation → Dealer Support → AXIS Closing）
  - Quick Questions（5 問）に基づく価値観・生活文脈・不安・負荷のスコアリングと「あなた理解 MAP」リアルタイム更新
  - Knowledge Graph 可視化（人 → 価値観 → 生活 → 負荷 → 体験 → 機能 → 候補車種）と Why Panel による推薦理由の説明
  - 最大 3 候補へのレコメンドと「なぜ外したか」の明示
- **差別化機能**
  - AI Delegation Level（Guide / Co-Pilot / Auto）による「AI にどこまで任せるか」の選択と UI 連動
  - 販売店向けインサイト＋提案トークスクリプトの自動生成（属人スキルの資産化）
  - 既存 PoC の Neo4j v3 オントロジー（4,205 件購入者レビュー）との接続によるデータ根拠
- **付加価値機能**
  - デモ用ナレーション自動切替（KG 画面）
  - KPI カード表示（商談品質・時間短縮・納得感など）

**3. 提供する価値**

- **ユーザー（来訪者・デモ体験者）**: 自分が主人公として理解され、AI に支配されず納得して選べる体験。選択肢の整理と理由の可視化。
- **ビジネス（OEM 本部）**: 説明可能 AI × KG の戦略ストーリー、販売店 DX・商談標準化への布石、AXIS（購入後体験）への自然な接続。
- **社会的価値**: 過剰な選択肢社会における判断負荷の軽減という、SDV・AI 時代の普遍的課題への応答。

**4. 実現方法の概要**

- **主要テクノロジー**: フロントエンド（デモ UI・KG 可視化・アニメーション）、API 層（セッション管理・スコアリング・Cypher クエリ）、Neo4j Knowledge Graph（v3 オントロジー）、必要に応じ LLM（トーク生成・補助説明）。
- **インテグレーションポイント**: 既存 `recommendation_engine` / `api_server` / `graph_builder`（Python + FastAPI + Neo4j）との接続。Quick Questions の回答を v3 Need / Capability / DecisionStyle にマッピング。
- **スケーラビリティ**: 本フェーズはショールームデモ（同時接続数・負荷は限定的）。将来は販売店端末・クラウド展開を想定し、API とグラフを分離した構成を維持する。

### 不足している情報（`docs/input` に未整備）

| カテゴリ | 不足内容 | 影響 |
|---------|---------|------|
| ビジネスモデル | 収益構造・正式な KPI 目標値・導入スコープ（デモのみか本番か） | 非機能要件の SLO・投資判断の精度が低下 |
| 制約条件 | 開発予算・納期・チーム構成・法務（個人情報・AI ガバナンス） | ロードマップ・リスク評価が仮定ベース |

### 置いた主要な仮定

1. **第一目的はショールームデモ**であり、本番販売店システムのフル機能はスコープ外（MVP は体験完走を優先）。
2. **同時利用者はデモ 1 セッション〜数名**（本部長向けプレゼン）。ピーク 50 同時接続は想定しない。
3. **車種データ・推薦ロジック**は既存 PoC（Honda 購入者レビュー 4,205 件、Neo4j v3）をベースに、デモ用に簡略マッピングで足りる。
4. **ブランド表記**はコンセプト上 Lexus Intelligence 系 UI も存在するが、デモ文言はメーカー中立または Honda データ前提で記載可能とする。
5. **AXIS 連携**は画面 7 でストーリー接続のみ（実 API 連携は将来フェーズ）。

## 技術スタック

### 推奨構成（デモモック向け・既存 PoC 活用）

| レイヤ | 技術 | 選定理由 |
|--------|------|----------|
| **フロントエンド** | **Next.js 15 (React) + TypeScript** | 7 画面のリッチアニメーション・KG 可視化（react-force-graph / D3 / Framer Motion）に適する。ショールーム品質の UI を Streamlit より実現しやすい。 |
| **KG 可視化** | react-force-graph または vis-network | ノード種別スタイル・エッジアニメーション要件に対応。 |
| **バックエンド API** | **FastAPI (Python)** — 既存 `api/api_server.py` を拡張 | 推薦エンジン・Neo4j ドライバ・LLM 連携が既に Python 実装。v3 対応済み `recommendation_engine` を再利用。 |
| **グラフ DB** | **Neo4j**（Neo4j Desktop ローカル or Aura） | v3 オントロジー構築済み（Consumer 4,205 / Need 55 / Capability 9 等）。説明可能パスの Cypher クエリがコア価値。 |
| **LLM** | Anthropic Claude API | 販売店トーク生成・補助説明。既存 `batch_llm_extractor` パターンと統一。 |
| **セッション** | Redis またはインメモリ（デモ時） | 軽量セッション。本番化時は Redis / Supabase 等を検討。 |
| **ホスティング（デモ）** | ローカル PC + 社内 Wi-Fi、または Vercel（FE）+ 社内 API | 企業 FW で Bolt 7687 ブロック時は Neo4j Desktop ローカル必須（実績あり）。 |

### 第一候補（プロンプト既定）との比較

| 項目 | Next.js + Supabase + Neon | 本推奨（Next.js + FastAPI + Neo4j） |
|------|----------------------------|-------------------------------------|
| 既存資産活用 | 低（再実装多め） | **高**（graph_builder / recommendation_engine 流用） |
| KG 表現 | RDB ではパス探索が弱い | **Neo4j が要件の中心** |
| デモ完成速度 | 中 | **高**（API・データは既存） |
| ショールーム UI | 高 | **高**（FE のみ新規） |

**結論**: フロントエンドはモダンスタック（Next.js + TypeScript + Vercel 可）、バックエンド・データ層は **既存 PoC（FastAPI + Neo4j）を維持・拡張**するハイブリッドが最適。

### 既存資産の再利用マップ

| 既存コンポーネント | モックでの役割 |
|-------------------|----------------|
| `graph_builder.py` / Neo4j v3 | KG Visualization の実データソース |
| `recommendation_engine.py` | Recommendation 画面のスコア・理由 |
| `api_server.py` | REST API 拡張（セッション・KG パス・トーク生成） |
| `sales_app.py` | 参考 UI・ロジック（本番デモ FE とは分離） |
| `consumer_decisions.json` | 類似事例・デモフォールバック |

## 開発ロードマップ

**フェーズ分けの根拠**: ①デモ完走の最短経路（MVP）、②説明可能性の核（KG 画面）、③OEM 向け差別化（Dealer・AXIS）、④本番化は別プロジェクト — の順でリスクを下げる。

### Phase 0：基盤整備（1〜2 週間）

- 既存 API の v3 整合性確認（推薦スコア 0% 問題の再発防止テスト）
- Quick Questions → Need/Capability マッピング仕様の確定
- デモ用ペルソナ JSON・オフラインフォールバックの整備
- Next.js プロジェクト雛形・デザインシステム（Calm / Premium）

**成果物**: 技術検証完了、API 契約（OpenAPI）ドラフト

### Phase 1：MVP デモ完走（3〜4 週間）

- 画面 1〜3（Opening / Questions / Delegation）+ 簡易 Recommendation（既存 API）
- デモモード（オフライン）で 7 分ルートの「通し」が可能
- 画面 5 の 3 候補・おすすめ度表示

**成果物**: 本部長向け内部リハーサル可能なビルド

### Phase 2：KG Visualization 主役（3〜4 週間）

- 画面 4 フル実装（グラフアニメーション + Why Panel + Neo4j  live 接続）
- 画面 5「なぜ外したか」・Delegation 連動
- ナレーション自動切替

**成果物**: コンセプトの核心が伝わるデモ

### Phase 3：OEM ストーリー完成（2〜3 週間）

- 画面 6 Dealer Support（トーク生成 + KPI）
- 画面 7 AXIS Closing
- プレゼン用オペレーターガイド・リセット手順

**成果物**: ショールーム本番デモ v1.0

### Phase 4：磨き込み・本番検討（任意）

- パフォーマンスチューニング、多言語、販売店パイロット設計
- Aura / クラウド Neo4j（FW 開放後）、認証・監査

## リスク分析

### 技術的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| Neo4j 企業 FW ブロック | クラウド KG が使えずデモ不能 | ローカル Neo4j Desktop + デモモード JSON の二重化（実績あり） |
| KG 可視化のパフォーマンス | 画面 4 が重くデモが止まる | ノード数上限・事前計算パス・モック fallback |
| v2/v3 オントロジー不整合 | スコア 0% 等の再発 | Cypher 回帰テスト、CI でグラフ整合チェック |
| LLM レイテンシ | Dealer 画面が遅延 | テンプレート先行表示 + ストリーミング更新 |

### ビジネス的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| 「ただのレコメンド」に見える | 差別化失敗 | 画面 4・6 を必ず見せるデモスクリプト |
| デモと本番のギャップ | 期待値乖離 | Closing で「本番化は Phase 4」と明示 |

## UI/UXの方向性

### デザインコンセプト

- **Calm Premium（落ち着いた高級感）**: 選択負荷を増やさない余白、抑制された配色、滑らかなモーション。
- **Explainable First**: グラフと Why Panel が主役。装飾より「理解の流れ」を優先。

### トーン＆マナー

- **プロフェッショナルかつ共感的**: 説教調にせず、「あなたを理解する」伴走感。
- **AI は支配しない**: Delegation Level で「自分で決める」尊厳を尊重。
- **日本語メイン**: 本部長向け。数値（%）は直感的に。

### ターゲットデバイス

- **PC / 大型タブレット / プロジェクター（1920×1080 以上）** がメイン。
- モバイルは Phase 4 以降。デモは横画面 2 カラム前提。

### ビジュアル要素

| 要素 | 方針 |
|------|------|
| 配色 | ネイビー基調 + ゴールドアクセント + 負荷ノードに控えめな赤 |
| タイポ | サンセリフ、見出しは細めウェイト、本文は可読性重視 |
| モーション | カード発光・エッジ Pulse・段階的グラフ接続（過剰アニメーションは避ける） |
| 既存参考 | 現行 `sales_app.py` の Lexus Intelligence Clean Design を踏襲しつつ、KG 画面は新規デザイン |

### 参考サイト／アプリ（方向性）

1. **Neo4j Bloom** — グラフ探索・説明可能性の UX 参考（https://neo4j.com/product/bloom/）
2. **Apple 製品構成ページ** — 選択の段階的絞り込み・Calm UI（https://www.apple.com/）
3. **Mercedes-Benz configurators** — プレミアム自動車の構成体験（ブランドトーン参考）
